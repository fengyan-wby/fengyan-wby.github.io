文献阅读：DSpark——面向高并发 LLM 服务的智能投机解码框架

本文由AI生成。

一、背景与动机

大语言模型（LLM）的推理本质上是一个自回归过程：每生成一个 token，就需要一次完整的前向传播，导致推理延迟与输出长度成正比。这在以下场景中尤为突出：

  • 实时对话助手：用户对响应速度极为敏感；
  • 多轮 Agentic 工作流：长链推理带来大量顺序等待。

投机解码（Speculative Decoding） 是目前最成熟的无损加速方案：用一个轻量级草稿模型（Draft Model）批量提议多个候选 token，再由目标模型（Target Model）并行验证，通过拒绝采样接受最长一致前缀并追加一个奖励 token。由于验证可以并行化，且严格保持目标分布不变，这一方案在不损失质量的前提下实现加速。

现有方法的两类草稿器及其缺陷

草稿器类型 代表方法 优点 缺点
自回归草稿器 Eagle3 建模词元间依赖，后缀质量稳定 草稿延迟与块大小成正比，架构浅层受限
并行草稿器 DFlash 单次前向即生成所有草稿，延迟近乎常数 各位置独立预测，后缀接受率急剧衰减

在此基础上，还存在一个系统层面的问题：无论草稿质量如何，对所有草稿 token 盲目验证会挤占目标模型的批处理容量，在高并发场景下严重拖累整体吞吐量。

论文将这两个瓶颈归纳为：一个在生成质量，一个在系统效率。DSpark 正是为同时解决这两个问题而设计的。

二、投机解码的理论基础

2.1 基本流程

每个解码周期：

  1. 草稿模型提议 \(\gamma\) 个候选 token \(x_1, \ldots, x_\gamma\)；
  2. 目标模型并行验证，逐位置计算接受概率：

\[\min\!\left(1,\; \frac{p^t_k(x_k)}{p^d_k(x_k)}\right)\]

  1. 在第一个被拒绝的位置截断，并由目标模型采样一个修正 token。

2.2 加速的三个杠杆

每生成 token 的平均延迟为：

\[L = \frac{T_{\text{draft}} + T_{\text{verify}}}{\tau}\]

其中 \(\tau\) 为每轮平均接受长度。提速的途径只有三条：

  • 降低 \(T_{\text{draft}}\)：让草稿生成更快；
  • 提高 \(\tau\)：让草稿质量更好；
  • 降低 \(T_{\text{verify}}\)：让验证更智能（不验无用 token）。

DSpark 的设计分别对应后两条杠杆。

三、DSpark 的架构设计

DSpark 由两个核心模块组成，其解码流程如下：

3.1 半自回归生成（Semi-Autoregressive Generation）

为什么纯并行会失败？

并行草稿器在预测时对同一块内的所有位置进行独立采样，无法感知前一个 token 的实际采样结果。这导致所谓的多模态碰撞（Multi-modal Collision）：

当上文允许"of course"和"no problem"两种延续时，并行模型可能输出"of problem"或"no course"，因为第 1 位置和第 2 位置各自在边缘分布上独立采样，而非联合采样。

实验证明，Eagle3（自回归）在各位置的条件接受率稳定甚至上升，而 DFlash（并行）则从首位置的高接受率快速下滑。

半自回归的解决方案

DSpark 将草稿生成分为两个阶段：

① 并行阶段（Parallel Stage）

使用 DFlash 作为并行骨干，一次前向传播产生所有位置的隐藏状态 \(h_1, \ldots, h_\gamma\) 和基础 logit \(U_1, \ldots, U_\gamma\)。

② 顺序阶段（Sequential Stage）

引入一个轻量级顺序模块，计算前缀依赖的转移偏置 \(B_k\)，使最终分布为：

\[p_k(v \mid x_0, x_{<k}) = \frac{\exp(U_k(v) + B_k(x_0, x_{<k}, v))}{\sum_{u \in \mathcal{V}} \exp(U_k(u) + B_k(x_0, x_{<k}, u))}\]

该分布是精确的 softmax，因此满足投机解码拒绝采样所需的精确逐 token 概率要求。

两种顺序模块实现

Markov Head（默认）

仅依赖前一个 token，使用低秩分解建模一阶转移偏置：

\[B(x_{k-1}, \cdot) = W_1[x_{k-1}]\, W_2 \in \mathbb{R}^V\]

其中 \(W_1 \in \mathbb{R}^{V \times r}\)，\(W_2 \in \mathbb{R}^{r \times V}\)，默认秩 \(r=256\)。参数量小，推理高效。

RNN Head

维护递归隐状态 \(s_k\) 以积累完整块内前缀信息：

\[s_k = \sigma(W_g z_k) \odot s_{k-1} + (1 - \sigma(W_g z_k)) \odot \tanh(W_c z_k)\]

\[B_k(x_{<k}, \cdot) = W_2^\top \tanh(W_o z_k)\]

其中输入 \(z_k = [s_{k-1};\, W_1[x_{k-1}];\, h_k]\)。RNN Head 能捕获更长的块内历史，但实现复杂度更高，在实验中仅带来边际增益，因此生产环境默认使用 Markov Head。

3.2 置信度调度验证（Confidence-Scheduled Verification）

3.2.1 置信度预测头

对每个草稿位置 \(k\)，置信度头输出一个标量 \(c_k \in (0,1)\)，表示在前 \(k-1\) 个 token 均被接受的条件下，第 \(k\) 个 token 被目标模型接受的概率：

\[c_k = \sigma\!\left(w^\top [h_k;\, W_1[x_{k-1}]]\right)\]

监督信号使用草稿分布与目标分布之间的总变差距离导出的解析接受率：

\[c^*_k = 1 - \frac{1}{2}\|p^d_k - p^t_k\|_1\]

3.2.2 后验校准：序列温度缩放（STS）

神经网络的置信度估计通常过于自信（overconfident）。由于硬件感知调度器需要利用置信度的绝对量级来估算期望接受长度，过高的原始置信度会导致调度决策失真。

DSpark 提出顺序温度缩放（Sequential Temperature Scaling, STS）：

  1. 位置 \(k\) 处的前缀接受概率为累积乘积 \(\prod_{i \leq k} c_i\)；
  2. 从左到右逐位置进行 1D 网格搜索，寻找最小化期望校准误差（ECE）的温度标量；
  3. 已校准位置的分数保持固定，不受后续位置校准影响。

实验表明，校准后各位置的 ECE 从原始的 3%–8% 降至约 1%，同时保持 ROC-AUC 在 0.81–0.90 的强判别能力。

3.2.3 硬件感知前缀调度器

核心思路：将验证长度选择建模为全局吞吐量最大化问题。

对一批 \(R\) 个并发请求，设请求 \(r\) 的位置 \(j\) 处的前缀存活概率为：

\[a_{r,j} = \prod_{i \leq j} c_{r,i}\]

目标模型的验证批大小为 \(B = \sum_{r=1}^R (1 + \ell_r)\)，系统期望接受 token 数为：

\[\tau = \sum_{r=1}^R \left(1 + \sum_{j=1}^{\ell_r} a_{r,j}\right)\]

令 \(\text{SPS}(B)\) 为以批大小 \(B\) 运行时的每步吞吐（steps/s，在引擎初始化时预测并存储），则优化目标为：

\[\max_{\{\ell_r\}} \;\Theta = \tau \cdot \text{SPS}(B)\]

由于 \(a_{r,j}\) 关于 \(j\) 单调非增，此优化问题允许一个贪心精确解：按全局 \(a_{r,j}\) 降序逐一加入候选 token，在吞吐 \(\Theta\) 不再增加时立即停止，算法可在线实时执行。

与静态阈值方法相比，硬件感知调度器的关键优势在于：

  • 低并发时：宽松验证预算（扩展至 4–6 个 token），充分利用空闲计算资源；
  • 高并发时：收紧验证预算，避免低置信度 token 占用关键批容量，防止吞吐崩溃。

四、训练目标

DSpark 的损失函数由三项加权组合：

\[\mathcal{L} = \alpha_{\text{ce}}\,\mathcal{L}_{\text{ce}} + \alpha_{\text{tv}}\,\mathcal{L}_{\text{tv}} + \alpha_{\text{conf}}\,\mathcal{L}_{\text{conf}}\]

默认权重为 \(\alpha_{\text{ce}} = 0.1\)，\(\alpha_{\text{tv}} = 0.9\)，\(\alpha_{\text{conf}} = 1.0\)，其中：

  • \(\mathcal{L}_{\text{ce}}\)：交叉熵损失；
  • \(\mathcal{L}_{\text{tv}}\)：基于总变差距离的分布匹配损失；
  • \(\mathcal{L}_{\text{conf}}\)：置信度头的监督损失。

五、实验结果

5.1 离线基准测试

在 Qwen3-{4B, 8B, 14B} 和 Gemma4-12B 四个目标模型上，覆盖数学推理、代码生成、日常对话三个领域：

目标模型 DSpark vs. Eagle3 DSpark vs. DFlash
Qwen3-4B +30.9% +16.3%
Qwen3-8B +26.7% +18.4%
Qwen3-14B +30.0% +18.3%

领域效应：结构化任务（数学、代码）接受长度天然更高（约 5.5），开放对话最低（约 3.5），这直接验证了动态调度的必要性。

5.2 架构分析

深度 vs. 性能：固定块大小为 7，DSpark 的接受长度随层数单调提升，且 2 层 DSpark 已超过 5 层 DFlash，体现了顺序建模的参数效率。

块大小 vs. 性能差距：随草稿块增大（\(\gamma\) 从 4 增至 16），DSpark 相对 DFlash 的优势持续扩大：

域 \(\gamma=4\) \(\gamma=16\)
数学 +16% +30%
代码 +15% +26%
对话 +18% +22%

延迟开销：顺序块的额外延迟极低。在批大小 128 下，将草稿长度从 4 扩展到 16，顺序块仅增加 0.2%–1.3% 的整轮延迟。

5.3 置信度头效果

在对话任务中，随置信度阈值提升，接受率从 45.7% 提升至 95.7%（大量低价值后缀被裁剪）；数学和代码任务中提升较温和（76.9%→92.5%，67.6%→92.0%），符合预期。

六、生产部署：DeepSeek-V4 在线评估

6.1 部署设置

  • 目标模型：DeepSeek-V4-Flash（preview）和 DeepSeek-V4-Pro（preview）
  • 草稿模型：DSpark-5（\(\gamma=5\)，Markov Head，3 层 MoE 并行骨干）
  • 基线：MTP-1（单 token 草稿，原生产基准）

MTP-1 之所以是基线，是因为在高并发下直接部署静态多 token 草稿（如 MTP-3/5）会因过度验证严重降低吞吐量，因此生产环境长期维持单 token 配置。DSpark 正是为了安全解锁多 token 草稿的性能潜力而设计。

6.3 负载自适应行为

动态调度器的行为符合设计预期：

  • 低并发：验证预算自动扩展，从 MTP-1 的静态 2 个 token 扩展至约 4–6 个；
  • 高并发：验证预算随负载平滑收缩，低置信度 token 被提前裁剪，防止资源争用。

七、局限性与未来方向

论文坦诚指出一个现有局限：DSpark 的并行骨干始终生成完整的 \(\gamma\) token 块，对于本身接受率极低的复杂查询，这部分草稿计算是不可回收的固定开销。

论文提出未来可探索基于难度感知的草稿模型早退出机制，使低价值请求绕过完整块生成，进一步降低无效草稿开销。

八、与相关工作的关系

方向 代表工作 DSpark 的位置
自回归草稿器 Eagle3、EAGLE DSpark 继承其依赖建模能力，但以更低开销实现
并行草稿器 DFlash、PARD、DART DSpark 以 DFlash 为骨干，追加顺序头解决后缀衰减
置信度调度 SpecTr、CASD DSpark 引入联合概率校准（STS）+ 硬件感知全局优化
非自回归生成 NAT、CRF-NAT、CTC-drafter DSpark 保持精确 softmax 概率，兼容标准拒绝采样

值得注意的是，论文提到并发工作 Domino 中的 CausalEncoder 与 RNN Head 在概念上类似，DFlare 则通过逐层融合改进条件瓶颈——这说明"在并行草稿上注入顺序依赖"这一方向已引发广泛关注。

九、总结与个人思考

DSpark 的核心洞察可以用一句话概括：

modern

并行草稿器的优势在于首 token 的高接受率（得益于更深的网络），自回归的优势在于后缀的一致性；只需一个极轻量的顺序模块，便可在不牺牲速度的前提下兼得两者之长。

这一思路的优雅之处在于：

  1. 架构侵入性极低：无需改动并行骨干，顺序头的参数量和计算量均可忽略不计；
  2. 概率精确性保留：不同于 CTC 等需要边际化的方法，Markov/RNN Head 输出的是精确 softmax 分布，直接兼容拒绝采样；
  3. 系统与算法协同设计：置信度头与硬件感知调度器的联合设计，使"草稿质量估计"真正落地为"系统吞吐收益"，而非仅停留在算法层面。

对于从事 LLM 推理加速研究的读者，DSpark 提供了一个非常实用的范本：生成质量与系统效率必须协同优化，孤立地提升草稿准确率或盲目延长验证序列，都无法在生产环境中真正发挥价值。
